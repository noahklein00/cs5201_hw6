Homework 6 - Complex Numbers, Outer Products, and Quantum Computing

Due on 2020-04-20 at class time

Purpose

The purpose of this assignment is to simulate a quantum computer by applying an object-oriented approach to the representation of quantum operators (gates). To achieve this you will apply your knowledge of the following:

  • Outer products
  • Complex numbers
  • Probability distributions
  • The “reduce” idiom
  • Templated functions and objects
  • Operator overloading
  • Interactions between objects of differing classes

Background

This assignment focuses on the simulation of a quantum computer and its application to the problem of function inversion, i.e. for some y=f(x)y=f(x), finding the value of xx that produces yy. With a classical computer it is clear that bruteforcing this requires NN evaluations of ff, where NN is the cardinality of the function’s domain. Quantum computing, and in particular the amplitude magnification enabled by Grover’s algorithm, allows for the function to be inverted in O(N−−√)O(N) time.

Program Specification

You are responsible for implementing a templated outer product - specifically, the Kronecker product. This class (or function) should overload its function evaluation operator to accept two matrices M1M1, M2M2 and should return a single matrix: M1⊗M2M1⊗M2 such that it is compatible with std::accumulate, called using iterators to one of your vectors of your matrices:

YourVector<YourMatrix<T>> list_of_operators;
auto composed_operator = std::accumulate(std::begin(list_of_operators),
        std::end(list_of_operators), YourMatrix<T>({1}), YourKronecker<T>());

You will likely find std::accumulate to be useful in the context of this assignment.

You will also implement a templated complex number class, which should implement the following:

  • Appropriate constructors and destructors
  • copy assignment
  • real and imag functions to return the real and imaginary parts of the complex number, respectively
  • stream operator (<<) to output a complex number - should be of the format a + bi, unless a or b is zero, in which case only bi or a should be displayed, respectively
  • overloaded unary - operator to return the additive inverse (of the real part, the imaginary part should be unchanged)
  • overloaded unary ! operator to return the complex conjugate
  • overloaded unary ~ operator to return the magnitude (recall that complex numbers are isomorphic to vectors in R2R2)
  • operators +, -, *, and their assignment counterparts

You should not use std::complex to implement this class. All arithmetic operations on this complex class should return new complex numbers.

You should also implement a quantum register class, which should be templated on the number of qubits. Note the use of a non-type template parameter and thus that the size of the register is known at compile time. Like a classical register, a quantum register stores the state of a set of bits - the only difference is that the bits are quantum in nature (i.e. qubits). What I’m looking to see in your code:

  • Encapsulation of quantum state for an arbitrary number of bits, using the format(s) discussed in lecture
  • A measurement operation that implements a weighted random choice, where the weights are the probabilities of potential discrete register states
  • An interface of your choice to allow for the application of gate(s), both “regular” and controlled, to the internal state of the register

In addition to appropriate constructors and destructors, your register should implement the << stream operator to measure and output the state of the system, which should be (on separate, labeled lines) the list of probabilities (in order from 00 to 2n−12n−1), and a space-separated binary string of 1s and 0s representing the measured state, with the most significant bit on the left.
For example, a qubit in the zero state must have the following output:

Probabilities: 1 0
Measured State: 0

Similarly, if the following circuit is simulated: The following output would be desired (the measured state will vary):

Probabilities: 0.5 0 0.5 0
Measured State: 1 0

Note that since measurement “collapses” the quantum system into a classical system, you should not allow the user to apply any quantum gates after measurement. Subsequent attempts by the user to measure the system should return the same result as the initial (“real”) measurement.

When measuring the state of a quantum register, you should use appropriate C++ random number utilities to simulate nondeterministic quantum behavior, i.e., to make a weighted random choice of one of the possible output states according to the register’s probability distribution.

The last set of classes you will implement are for the gates themselves. The object-oriented design decisions you make when implementing these classes should take into account the existence of controlled gates, which are by definition implemented in terms of more basic gates.
For example, a controlled not (CNOT) gate is implemented in terms of a NOT gate, and a controlled controlled not with two control bits (CCNOT) is implemented in terms of a CNOT gate. Also consider the usefulness of inheritance and polymorphism when implementing your gates.
What I’m looking to see in your code:

  • Encapsulation of unitary operators, represented as matrices
  • Functionality that implements the linear algebra techniques used in lecture to compose gates and also to create controlled gates (you may delegate this to your quantum register class if you wish)

Controlled gates must support at least one and two control bit configurations (the minimum required to implement the circuits displayed below). To help you implement controlled gates, I have gone through another example.

In addition to whatever appropriate constructors you find necessary, you should ensure that your gate class(es) include a constructor that accepts a matrix representation of a gate.

Neither your gates nor your quantum registers must support arbitrary (templated) internal state representations - you may assume floating-point scalar instantiations of your complex number class.

You must configure your gates and registers such that gates can be applied to individual qubits in a register. This “apply” functionality is the key requirement of your gates. When controlled gates are applied, the user should be able to specify one or more bits with which the gate should be controlled. One way of implementing this might look like: qreg.apply(2, controlled_not, {0,1}), which would apply a not gate to qubit 2 controlled by qubits 0 and 1. For an additional challenge, you might try to replicate the syntax used by quantum assembly languages, for example, qreg[2] | controlled_not({0,1}). Any reasonable syntax for expressing the application of gates to a quantum system is acceptable.

Testing Your Code

Some ideas for testing your gates and registers are as follows:

  • Ensure that applying a Hadamard gate to a single-bit register results in the state presented in lecture, and a measurement with equal probability of 0 and 1
  • Ensure that applying a NOT gate to a single-bit register inverts the state
  • Ensure that applying a NOT gate to both bits in a two-bit register inverts the state
  • Try to create a controlled NOT gate, and ensure that the 4×44×4 linear operator that is generated matches what was presented in lecture
  • Repeat the above, except for a controlled controlled NOT gate (a gate with two control qubits)
  • Once both of the above work, try swapping the positions of the control and target bits, making sure that everything still works

Making sure that the algorithms produce the correct result is fairly simple:

  • The result of Grover’s algorithm (the first circuit) should always be one of the marked states
  • The result of Shor’s algorithm (the second circuit) should always be the period of the function f(x)=ax(modN)f(x)=ax(modN) - you can graph f(x)f(x) to verify

If you are having trouble integrating your gates into a circuit, consider using Quirk in an iterative fashion. That is, apply one gate to both the Quirk simulation and your simulation, verify that the resulting states match, then repeat for all gates in a circuit.
